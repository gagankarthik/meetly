# Deployment Guide

Meetly has three deployable artifacts:

1. **Desktop application** (Electron, Win + macOS) — what the user installs
2. **AWS infrastructure** (Terraform) — Cognito, DynamoDB, KMS, IAM
3. **Marketing site** (`landing/`) — static HTML, deployable anywhere

This guide covers all three from a clean checkout to a public release.

---

## 0. Prerequisites

| Tool | Version | Purpose |
|---|---|---|
| Node.js | ≥ 20.18 | Build + runtime |
| npm | ≥ 10 | Package manager |
| Terraform | ≥ 1.5 | Infra provisioning |
| AWS CLI | ≥ 2 | Optional, for verification |
| An AWS account with admin perms | — | For the first `terraform apply` |

> The Electron app itself runs with **non-admin Cognito Identity Pool credentials** at runtime. Admin keys are only needed at provisioning and for the dev "auto-confirm signup" shortcut.

## 1. AWS infrastructure (one-time per environment)

### 1.1 Provision

```sh
cd infra
terraform init
terraform apply -var region=us-east-2
```

What gets created:

| Resource | Purpose |
|---|---|
| `aws_kms_key.meetly` | Customer-managed key encrypting DynamoDB at rest |
| `aws_dynamodb_table.meetly` | Single-table — PK/SK schema, PAY_PER_REQUEST, PITR on |
| `aws_cognito_user_pool.main` | Email + password authentication |
| `aws_cognito_user_pool_client.desktop` | Public client (no secret) for the desktop app |
| `aws_cognito_identity_pool.main` | Vends temporary AWS creds to signed-in users |
| `aws_iam_role.authenticated` + policy | Per-user scoped `dynamodb:LeadingKeys = USER#<sub>` |

### 1.2 Capture outputs

```sh
terraform output env_template
```

This prints exactly the lines you need to paste into `.env.local`:

```
AWS_REGION=us-east-2
COGNITO_USER_POOL_ID=us-east-2_XXXXXXXXX
COGNITO_APP_CLIENT_ID=xxxxxxxxxxxxxxxxxxxxxxxxxx
COGNITO_IDENTITY_POOL_ID=us-east-2:xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx
DYNAMODB_TABLE=Meetly
```

### 1.3 Verify

```sh
aws dynamodb describe-table --table-name Meetly --region us-east-2 --query 'Table.TableStatus'
# → "ACTIVE"

aws cognito-idp describe-user-pool --user-pool-id us-east-2_XXXXXXXXX --region us-east-2 --query 'UserPool.Name'
# → "meetly-prod-users"
```

### 1.4 Tear down (when needed)

```sh
cd infra
terraform destroy -var region=us-east-2
```

> ⚠️ This wipes the DynamoDB table (including all user data) and the Cognito pool. Take a backup first if you need to keep anything.

## 2. Configure environment

Copy `.env.example` to `.env.local` and fill in:

```env
# API keys (per-user override available in Settings)
DEEPGRAM_API_KEY=...
OPENAI_API_KEY=...

# AWS (from terraform output env_template)
AWS_REGION=us-east-2
COGNITO_USER_POOL_ID=us-east-2_XXXXXXXXX
COGNITO_APP_CLIENT_ID=xxxxxxxxxxxxxxxxxxxxxxxxxx
COGNITO_IDENTITY_POOL_ID=us-east-2:xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx
DYNAMODB_TABLE=Meetly

# Admin IAM keys — required only to auto-confirm signups (skips OTP email)
# Generate an IAM user with cognito-idp:AdminConfirmSignUp permission and paste here.
AWS_ADMIN_ACCESS_KEY_ID=AKIA...
AWS_ADMIN_SECRET_ACCESS_KEY=...

# Dev-only auth bypass (DO NOT ship)
DEV_SKIP_AUTH=false
```

> `.env.local` is gitignored by default. **Never commit it.**

## 3. Desktop app — build & distribute

### 3.1 Dev run

```sh
npm install
npm run dev
```

Auth window opens first. Sign up → auto-confirmed via admin keys (no OTP) → Hub opens.

### 3.2 Production build

```sh
# Windows installer (NSIS .exe)
npm run dist:win

# macOS DMG (Universal Intel + Apple Silicon)
npm run dist:mac

# Both
npm run dist
```

Artifacts land in `release/<version>/`:

```
release/0.1.0/
├── Meetly-0.1.0-win-x64.exe
├── Meetly-0.1.0-mac-arm64.dmg
└── Meetly-0.1.0-mac-x64.dmg
```

### 3.3 App icon

`resources/icon.png` is generated by `scripts/generate-icon.cjs`. Regenerate if you change the brand:

```sh
node scripts/generate-icon.cjs
```

For production-quality icons, replace with proper `.ico` (Windows) and `.icns` (macOS) assets and update `electron-builder` config in `package.json`.

### 3.4 Code-signing (production)

- **Windows**: provide `CSC_LINK` + `CSC_KEY_PASSWORD` env vars pointing to your code-sign cert. Without a signed binary, Windows SmartScreen warns users.
- **macOS**: requires an Apple Developer account. Set `APPLE_ID`, `APPLE_APP_SPECIFIC_PASSWORD`, and `APPLE_TEAM_ID`. The build will codesign + notarize via electron-builder.

### 3.5 Distribution channels

| Channel | How |
|---|---|
| Direct download | Host `.exe` / `.dmg` on the landing site or S3 + CloudFront |
| Auto-update | Configure `electron-updater` against an S3 bucket or GitHub Releases (not enabled by default — add to `package.json` `build.publish`) |
| Microsoft Store / Mac App Store | Requires extra packaging steps; out of scope for first release |

## 4. Landing page (`landing/`)

The marketing site is a single static `index.html` — Tailwind via CDN, no build step.

### 4.1 Local preview

```sh
cd landing
python -m http.server 4000   # or: npx serve
# open http://localhost:4000
```

### 4.2 Deploy options (pick one)

#### GitHub Pages (recommended — zero config after enable)
The repo ships with `.github/workflows/pages.yml`. On every push to `main` that
touches `landing/**`, it bundles the folder and deploys to Pages via the
official `actions/deploy-pages` action.

One-time setup:
1. Push the repo to GitHub.
2. Repo Settings → Pages → **Source: GitHub Actions**.
3. Push any change under `landing/` (or run the workflow manually under Actions → "Deploy landing page to GitHub Pages" → Run workflow).
4. Site appears at `https://<user>.github.io/<repo>/`.

For a custom domain, add a `landing/CNAME` file with the bare domain and configure DNS as GitHub instructs.

#### Netlify
```sh
npx netlify-cli deploy --dir=landing --prod
```

#### Vercel
```sh
npx vercel --cwd landing --prod
```

#### AWS S3 + CloudFront
```sh
aws s3 sync landing s3://meetly-landing/ --delete
aws cloudfront create-invalidation --distribution-id <id> --paths "/*"
```

### 4.3 Custom domain

Point `meetly.app` (or your domain) at your hosting target:
- Netlify/Vercel: set the domain in dashboard, follow DNS instructions
- CloudFront: set Alternate Domain Name, attach an ACM certificate, add a CNAME

## 5. CI / CD (suggested)

A minimal GitHub Actions workflow you can drop into `.github/workflows/release.yml`:

```yaml
name: release
on:
  push:
    tags: ['v*']
jobs:
  build:
    strategy:
      matrix:
        os: [windows-latest, macos-latest]
    runs-on: ${{ matrix.os }}
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '20' }
      - run: npm ci
      - run: npm run dist
        env:
          CSC_LINK: ${{ secrets.CSC_LINK }}
          CSC_KEY_PASSWORD: ${{ secrets.CSC_KEY_PASSWORD }}
          APPLE_ID: ${{ secrets.APPLE_ID }}
          APPLE_APP_SPECIFIC_PASSWORD: ${{ secrets.APPLE_APP_SPECIFIC_PASSWORD }}
          APPLE_TEAM_ID: ${{ secrets.APPLE_TEAM_ID }}
      - uses: actions/upload-artifact@v4
        with:
          name: ${{ matrix.os }}-installer
          path: release/**/*.{exe,dmg}
```

The landing page deploys separately on every push to `main`:

```yaml
name: landing
on:
  push:
    branches: [main]
    paths: ['landing/**']
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: amondnet/vercel-action@v25
        with:
          vercel-token: ${{ secrets.VERCEL_TOKEN }}
          vercel-org-id: ${{ secrets.VERCEL_ORG_ID }}
          vercel-project-id: ${{ secrets.VERCEL_PROJECT_ID }}
          working-directory: landing
          vercel-args: '--prod'
```

## 6. Release checklist

Before tagging a release:

- [ ] `npm run typecheck` passes
- [ ] `npm run lint` passes
- [ ] `npm run dist` succeeds on both Win + Mac
- [ ] `DEV_SKIP_AUTH=false` in shipped builds (never `true` in production)
- [ ] Admin IAM keys are **not** baked into the shipped binary (env is read at runtime; ship without them)
- [ ] Test sign-up + sign-in end-to-end with a fresh email
- [ ] Test a full meeting: start → talk → stop → summary appears in Library
- [ ] Test Chat tab in Library against an existing meeting
- [ ] Test offline behavior (start with no network → friendly error)
- [ ] Update `version` in `package.json` and tag in git
- [ ] Landing page links to the new installers

## 7. Operational notes

- **Costs at idle**: KMS CMK ≈ $1/month. DynamoDB PAY_PER_REQUEST → $0 when no usage. Cognito → free up to 50k MAU.
- **Logs**: each window writes to `%APPDATA%/Meetly/logs` (Windows) or `~/Library/Logs/Meetly` (macOS).
- **Wiping a user's data**: `aws dynamodb query --table-name Meetly --key-condition-expression "PK = :pk" --expression-attribute-values '{":pk":{"S":"USER#<sub>"}}'` then batch-delete.
- **Rotating Cognito client secret**: not applicable — desktop client is public (`generate_secret = false`).
